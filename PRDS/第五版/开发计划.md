# 第五版开发计划
## 1. 当前状态
状态：已完成

当前第五版从第四版正式骨架继续推进，目标不再是扩张功能，而是把已有后端能力收束为 `v1.0` 可部署版本。

## 2. 开发目标
在第四版已有实现基础上完成真实可部署闭环，包括：

1. 服务端真实运行
2. MySQL / Redis 真实链路
3. 搜索重建真实写入
4. 统计、任务、同步最小闭环
5. 部署配置与 `v1.0` 文档收尾

## 3. 开发策略
第五版按以下策略推进：

1. 先同步文档状态，再开始实现
2. 先解决运行和配置问题，再谈功能补强
3. 优先保证后端链路真实可用，不追求额外平台能力
4. 优先保证部署清晰、维护简单，不引入新的复杂基础设施
5. 范围调整、数据库结构变化、接口边界变化、部署方式变化必须先更新文档

## 4. 阶段划分
### 阶段一：文档收束与上线边界确认

目标：
- 创建第五版总文档与模块文档
- 明确 `v1.0` 必做项与明确不做项
- 明确部署、环境变量、内容目录规则

状态：已完成

### 阶段二：服务端真实运行与配置打通

目标：
- 修复本地服务启动与运行问题
- 让服务进程正确读取 MySQL / Redis / SITE_URL 配置
- 确认后台页能展示真实运行态

状态：已完成

### 阶段三：搜索、统计、任务、同步真实闭环

目标：
- 完成真实搜索重建
- 生成统计快照
- 写入任务记录
- 写入同步记录
- 验证 `/admin/search`、`/admin/stats`、`/admin/tasks`、`/admin/sync`

状态：已完成

### 阶段四：部署配置与 v1.0 收尾

目标：
- 整理本地运行和生产部署说明
- 整理服务器准备清单
- 补齐 VERSION_NOTES 与模块状态文档
- 输出是否达到 `v1.0 可部署` 的判断

状态：已完成

## 5. 优先顺序
第五版按以下顺序推进：

1. 先修复并确认服务端真实运行问题
2. 再跑通 `/admin/search`、`/admin/stats`、`/admin/tasks`、`/admin/sync`
3. 再完成真实搜索重建、任务记录、同步记录写入
4. 再整理部署配置、环境变量、运行命令
5. 最后输出 `v1.0` 部署说明和收尾结论

## 6. 前置条件与基础设施
第五版默认需要以下基础设施，但不由代码自动创建：

1. MySQL
2. Redis
3. 服务器上的 Rust 运行或构建环境
4. 反向代理
5. 正式域名或至少可访问的站点地址

当前已确认：
- MySQL：已创建 `my-blog` 库
- Redis：已启动并可用

## 7. 完成判断
第五版可以认为完成，当以下条件同时满足：

1. 第五版文档已建立并与真实状态同步
2. 服务端可以按文档稳定启动
3. MySQL 与 Redis 已完成真实链路验证
4. 搜索重建、统计、任务、同步都留下正式记录
5. 用户可以根据部署文档自行准备服务器并上线
6. 可以明确回答哪些内容进入 `v1.0`，哪些被刻意留在 `v1.0` 之外
