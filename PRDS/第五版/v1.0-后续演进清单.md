# 第五版 - v1.0 后续演进清单

这份文档只记录 `v1.0` 明确留在版本外、但后续值得继续推进的事项，方便在不打乱当前部署版本边界的前提下继续演进。

## 1. 文档状态
- 状态：待后续版本逐项认领
- 适用阶段：`v1.0` 发布之后
- 原则：先保证现有站点稳定运行，再决定是否继续扩张能力

## 2. 后续功能候选
### 2.1 内容治理与后台能力
1. 内容后台增加更细的质量检查规则，例如摘要缺失、标签异常、系列断链、日期异常。
2. 后台增加内容筛选保存、批量校验、问题分组视图。
3. 内容详情页增加更清晰的内容关系图或引用链路展示。
4. 为搜索重建、统计刷新、同步执行补充更明确的失败原因与重试提示。

### 2.2 搜索与索引增强
1. 提升搜索排序策略，例如标题权重、标签权重、摘要命中权重。
2. 增加搜索结果高亮、匹配片段和更稳定的查询诊断说明。
3. 为搜索索引增加更明确的增量更新方案，减少每次全量重建成本。
4. 为搜索重建增加更完整的错误日志与告警输出。

### 2.3 统计与任务系统增强
1. 为统计页增加更长周期的趋势视图，但仍应服务站点治理，不必升级成 BI 平台。
2. 为任务记录增加筛选、失败重试、任务详情页。
3. 如果后续站点任务明显增多，再评估是否拆分独立 worker。
4. 为统计、搜索、同步任务建立更稳定的告警和巡检机制。

### 2.4 同步能力增强
1. 把当前最小同步闭环升级为真实导入器，而不只是站内数据写入验证。
2. 为每类同步源定义更清晰的数据映射和去重规则。
3. 增加同步结果对内容页的反向校验，避免“写入成功但展示不完整”。
4. 如果后续真的需要第三方平台接入，再单独立项，不直接塞进当前版本。

## 3. 后续内容建设
### 3.1 笔记内容
1. 持续补充 `content/notes/*.md`，优先填充 Rust、C++、Bochs 三类技术板块。
2. 给已有笔记补齐标签、摘要、发布日期和系列关系，减少后台问题项。
3. 把实验记录整理成更适合公开阅读的结构，而不只是零散备忘。

### 3.2 博客内容
1. 持续在 `content/blog/*.md` 发布正式文章，沉淀阶段性总结。
2. 补充对项目设计、部署过程、Rust 学习路径的长文说明。
3. 为旧文章统一检查封面文案、摘要风格、标签一致性。

### 3.3 项目内容
1. 在 `content/projects/*.md` 持续维护项目背景、时间线、结果与复盘。
2. 为每个项目补充更明确的状态定义，例如进行中、维护中、暂停中、已归档。
3. 如果项目数量明显增加，再评估是否需要独立的项目索引字段或专题页。

## 4. 基础设施与运维候选
1. 为服务器部署补充自动备份策略，至少覆盖 MySQL、Redis 和 `content/` 目录。
2. 为生产环境增加日志轮转、服务重启策略和健康检查。
3. 增加部署后巡检脚本，自动验证首页、搜索后台、统计后台、同步后台是否可用。
4. 如果后续有域名与 HTTPS，上线前补充证书续期与反向代理检查流程。

## 5. 刻意不进入 v1.0 的内容
以下内容明确不属于当前 `v1.0`，后续若要做，应单独评估并重新立项：

1. 完整用户系统
2. 多角色权限与复杂 RBAC
3. 评论社区
4. 面向外部的开放平台 API
5. 复杂工作流编排
6. 消息队列、分布式调度、独立任务中心
7. 大规模视觉重设计

## 6. 后续执行建议
建议按以下顺序继续推进：

1. 先持续补内容，把 `content/blog`、`content/notes`、`content/projects` 做厚。
2. 再增强后台治理能力，减少内容缺项与手工核对成本。
3. 最后再评估是否值得继续扩张搜索、同步或任务系统。
