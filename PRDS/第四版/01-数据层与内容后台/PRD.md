# 第四版 - 数据层与内容后台 PRD
## 1. 功能定位
这个模块负责把第三版的静态内容聚合，推进为第四版的正式内容数据层与后台治理入口。

它解决的问题是：
- 内容越来越多后，单靠构建期聚合不够稳定
- 内容治理缺少统一后台入口
- 后续搜索、统计、同步都需要正式数据边界

## 2. 当前版本状态
状态：进行中

当前判断：
- 已正式开始第四版第一项
- 第一批实现以“第三版 Markdown 内容源 -> 第四版后台读模型”的桥接方式推进
- 当前不直接实现数据库写入，但会先把数据边界、后台边界、接口边界定出来

## 3. 当前实现范围
本模块第四版应覆盖：

1. 内容实体模型
2. 标签、系列、归档、关联的正式数据关系
3. 内容状态字段
4. 后台内容列表与详情骨架
5. 服务端内容查询与聚合接口
6. 第三版现有内容系统的衔接层

当前第一批实现的具体边界：
- 统一内容类型：`blog`、`notes`、`projects`
- 统一后台内容 ID：使用“内容类型 + slug”的派生标识，供 `/admin/content/:id` 使用
- 统一后台状态字段：先提供可演进的状态字段与问题摘要，不提前进入复杂审核流
- 统一来源字段：明确内容当前仍来自仓库内 Markdown 文件
- `notes` 将补充正式 `board` 字段，用于区分 `rust`、`cpp`、`bochs` 等板块，并供后台与搜索索引复用
- 笔记内容源当前固定放在 `content/notes/*.md`，新增笔记要通过 front matter 写入 `board`

## 4. 后端边界说明
这个模块已经明确触及第四版后端边界：

1. 数据库 schema
2. 服务端读写逻辑
3. 后台管理入口

当前阶段的实际边界说明：
- 读模型、聚合模型、后台视图模型会先落在服务端代码中
- 数据库 schema 只在文档和结构层面预留，不在本轮直接创建
- 写入逻辑只预留接口骨架，不在本轮落地实际持久化

## 5. 暂不要求的内容
1. 完整多角色权限
2. 富文本在线编辑器
3. 复杂审核流
4. 面向外部用户的内容协作
5. 真正数据库迁移与生产级后台写入

## 6. 当前实现结果
1. 已建立统一后台内容类型：`blog`、`notes`、`projects`
2. 已建立统一后台内容 ID：`{content_type}--{slug}`
3. 已建立后台列表项、详情项、问题摘要、概览统计等读模型
4. 已提供 `/admin`、`/admin/content`、`/admin/content/:id`
5. 已提供内容后台概览、列表、详情的服务端查询接口
6. 已确认当前后台数据直接桥接第三版现有内容解析与关联逻辑
7. 已为 `notes.board` 建立正式内容字段边界，供笔记板块与搜索索引复用
8. 已明确后续 Rust / C++ / Bochs / 通用笔记都继续从 `content/notes/*.md` 生效
