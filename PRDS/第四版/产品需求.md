# 第四版产品需求
## 1. 第四版定位
第四版的目标，不再只是增强静态内容组织，而是把这个博客从“以 Markdown 为核心的内容站点”推进到“具备最小后端能力的个人内容平台”。

第三版已经完成：
- 个人主页与公开导航骨架
- 标签、归档、系列、内容关联
- 搜索升级
- 内容模型增强与发布前治理

第四版要解决的问题是：
- 内容量继续增长后，静态聚合与静态校验会越来越重
- 搜索、统计、关联与同步能力缺少持久化基础
- 内容治理仍然偏“提交代码前检查”，缺少后台化管理入口
- 后续如果要做自动化、外部同步、任务编排，第三版的结构已经不够

## 2. 第四版总体策略
第四版建议采用“后端能力最小闭环”策略：

1. 先建立正式数据层和后端服务边界
2. 先做内容后台与治理入口，再做更复杂的自动化
3. 先让搜索、统计、同步具备基础设施，再逐步增强体验
4. 保持个人博客属性，不在第四版直接扩张为社区平台

第四版会正式进入后端边界，明确包含：
- 数据库
- 后端服务
- 后台管理入口
- 持久化搜索索引或索引元数据
- 持久化统计
- 任务系统
- 外部数据同步接口

## 3. 第四版目标
第四版需要完成以下目标：

1. 建立内容平台的数据层与服务端结构
2. 建立最小后台内容管理与治理能力
3. 让搜索、统计、关联具备持久化和可演进基础
4. 为外部同步、自动任务和开放接口建立边界

## 4. 第四版范围
### 4.1 必做范围

1. 数据层与内容后台
2. 搜索与索引基础设施
3. 统计与任务系统
4. 外部同步与开放能力基础

### 4.2 建议增强

1. 后台中的内容状态面板
2. 内容审核与发布流转状态
3. 搜索重建任务与统计重算任务
4. GitHub、RSS、外部内容源的统一同步入口
5. 面向后续版本的 API 认证预留

## 5. 模块范围说明
### 5.1 数据层与内容后台

第四版需要把第三版的内容聚合结构推进为正式内容数据层，至少包括：
- 内容表述模型
- 标签、系列、归档、关联的正式结构
- 内容状态字段
- 后台查询与编辑入口

这里已经明显触及第四版后端边界：
- 数据库 schema
- 后台管理界面
- 服务端写入逻辑

### 5.2 搜索与索引基础设施

第四版不再只依赖构建期静态搜索结果，而是需要建立：
- 可重建的搜索索引流程
- 可持久化的索引数据或索引元信息
- 服务端搜索接口边界

### 5.3 统计与任务系统

第四版需要从“轻量静态统计”升级到“最小持久化统计”：
- 访问统计
- 内容热度或最近活跃统计
- 重建、同步、统计归档等后台任务入口

这部分同样明确属于第四版后端能力，不再属于第三版前端展示层增强。

### 5.4 外部同步与开放能力

第四版建议建立：
- 外部资料源同步边界
- GitHub 或其他来源的内容/项目数据导入流程
- 面向第五版继续扩展的开放接口结构

## 6. 页面与功能范围
第四版正式实现重点建议为：

1. `/admin` 后台入口
2. `/admin/content`
3. `/admin/content/:id`
4. `/admin/search`
5. `/admin/stats`
6. `/admin/tasks`
7. `/admin/sync`
8. 服务端内容接口、搜索接口、统计接口、任务接口

并对以下已有能力继续升级：

1. `/me`
2. `/search`
3. `/blog/:slug`
4. `/notes/:slug`
5. `/projects/:slug`

## 7. 第四版不做
第四版仍然明确不做以下内容：

1. 完整用户社区
2. 评论系统
3. 多租户能力
4. 复杂 RBAC 多角色权限
5. 完整工作流编排平台
6. 大规模外部平台分发矩阵

## 8. 依赖基础设施说明
如果第四版继续推进实现，后续大概率需要用户自行准备以下基础设施：

1. 数据库
2. 缓存
3. 对象存储
4. 队列或任务执行基础设施

推荐原则：
- 数据库优先用于内容元数据、搜索元数据、统计与任务记录
- 缓存优先用于搜索结果、聚合视图和热点统计
- 对象存储只在后续需要上传资源时引入
- 队列用于同步任务、索引重建、统计归档等异步场景

## 9. 成功标准
第四版完成后，应满足以下标准：

1. 内容系统不再完全依赖静态构建时聚合
2. 后台可以查看和治理内容、索引、统计、任务状态
3. 搜索、统计、同步具备正式服务端边界
4. 第五版可以在第四版基础上继续做用户能力、开放能力或更复杂自动化
