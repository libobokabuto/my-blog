# 第四版产品需求
## 1. 第四版定位
第四版的目标，不再只是增强静态内容组织，而是把这个博客从“以 Markdown 为核心的内容站点”推进到“具备最小后端能力的个人内容平台”。

第三版已经完成：
- 个人主页与公开导航骨架
- 标签、归档、系列、内容关联
- 搜索升级
- 内容模型增强与发布前治理

第四版要解决的问题是：
- 内容量继续增长后，静态聚合与静态校验会越来越重
- 搜索、统计、关联与同步能力缺少持久化基础
- 内容治理仍然偏“提交代码前检查”，缺少后台化管理入口
- 后续如果要做自动化、外部同步、任务编排，第三版的结构已经不够

## 2. 第四版总体策略
第四版建议采用“后端能力最小闭环”策略：

1. 先建立正式数据层和后端服务边界
2. 先做内容后台与治理入口，再做更复杂的自动化
3. 先让搜索、统计、同步具备基础设施，再逐步增强体验
4. 保持个人博客属性，不在第四版直接扩张为社区平台

第四版会正式进入后端边界，明确包含：
- 数据库
- 后端服务
- 后台管理入口
- 持久化搜索索引或索引元数据
- 持久化统计
- 任务系统
- 外部数据同步接口

## 3. 第四版目标
第四版需要完成以下目标：

1. 建立内容平台的数据层与服务端结构
2. 建立最小后台内容管理与治理能力
3. 让搜索、统计、关联具备持久化和可演进基础
4. 为外部同步、自动任务和开放接口建立边界

当前状态：进行中

当前已确认第四版采用“第三版 Markdown 内容源 + 第四版服务端读模型 + MySQL/Redis 基础设施”的过渡方式推进，而不是在第一步就直接切到数据库写入。

## 4. 第四版范围
### 4.1 必做范围

1. 数据层与内容后台
2. 搜索与索引基础设施
3. 统计与任务系统
4. 外部同步与开放能力基础

### 4.2 建议增强

1. 后台中的内容状态面板
2. 内容审核与发布流转状态
3. 搜索重建任务与统计重算任务
4. GitHub、RSS、外部内容源的统一同步入口
5. 面向后续版本的 API 认证预留

## 5. 模块范围说明
### 5.1 数据层与内容后台

第四版需要把第三版的内容聚合结构推进为正式内容数据层，至少包括：
- 内容表述模型
- 标签、系列、归档、关联的正式结构
- 内容状态字段
- 后台查询与编辑入口

第一批正式实现优先落地：
- 后台读模型与聚合模型
- `/admin`、`/admin/content`、`/admin/content/:id` 页面骨架
- 服务端内容查询与聚合接口骨架
- 第三版现有 `blog / notes / projects` 内容系统的衔接层

这一批先不直接落地数据库写入，但接口命名、数据结构和后台视图需要按未来可切换到数据库的方式设计。

`notes` 已补充正式 `board` 字段，用于区分 `rust`、`cpp`、`bochs`、`general` 等笔记板块，并作为后台治理和搜索索引的正式字段之一。

当前笔记内容源约定：
- 笔记文件继续放在 `content/notes/*.md`
- front matter 至少包含 `title`、`summary`、`date`、`board`、`stage`、`source`、`experiment_state`、`tags`
- `board` 当前正式值为 `rust`、`cpp`、`bochs`、`general`

### 5.2 搜索与索引基础设施

第四版不再只依赖构建期静态搜索结果，而是需要建立：
- 可重建的搜索索引流程
- 可持久化的索引数据或索引元信息
- 服务端搜索接口边界

当前这一阶段优先采用：
- MySQL 持久化搜索文档与重建记录
- Redis 记录重建状态、轻量诊断状态和后续任务预留键空间
- 与第三版现有 `/search` 前台入口兼容的过渡搜索结构

索引文档字段至少需要覆盖：
- 内容类型、slug、标题、摘要、标签
- `notes.board`、`notes.stage`
- `blog.series`
- `projects.status`
- 正文纯文本与重建时间

### 5.3 统计与任务系统

第四版需要从“轻量静态统计”升级到“最小持久化统计”：
- 统计快照
- 内容治理指标
- 重建、同步、统计归档等后台任务入口

当前第一批正式实现已经包含：
- `stats_snapshots` 持久化结构
- `task_runs` 统一任务记录结构
- `/admin/stats`
- `/admin/tasks`

### 5.4 外部同步与开放能力

第四版建议建立：
- 外部资料源同步边界
- GitHub 或其他来源的内容/项目数据导入流程
- 面向第五版继续扩展的开放接口结构

当前第一批正式实现已经包含：
- `sync_sources` 与 `sync_runs` 结构
- `/admin/sync`
- 同步源占位模型与手动触发入口骨架

## 6. 页面与功能范围
第四版正式实现重点建议为：

1. `/admin`
2. `/admin/content`
3. `/admin/content/:id`
4. `/admin/search`
5. `/admin/stats`
6. `/admin/tasks`
7. `/admin/sync`
8. 服务端内容接口、搜索接口、统计接口、任务接口、同步接口

当前已经进入正式实现的页面与接口：
1. `/admin`
2. `/admin/content`
3. `/admin/content/:id`
4. `/admin/search`
5. `/admin/stats`
6. `/admin/tasks`
7. `/admin/sync`

## 7. 第四版不做
第四版仍然明确不做以下内容：

1. 完整用户社区
2. 评论系统
3. 多租户能力
4. 复杂 RBAC 多角色权限
5. 完整工作流编排平台
6. 大规模外部平台分发矩阵

## 8. 依赖基础设施说明
如果第四版继续推进实现，后续大概率需要用户自行准备以下基础设施：

1. MySQL
2. Redis
3. 对象存储
4. 队列或任务执行基础设施

推荐原则：
- MySQL 优先用于内容元数据、搜索元数据、统计与任务记录
- Redis 优先用于搜索运行态、轻量状态缓存和后续任务预留键空间
- 对象存储只在后续需要上传资源时引入
- 队列用于同步任务、索引重建、统计归档等异步场景

当前判断：
- MySQL 与 Redis 已足够支撑第四版当前骨架
- 对象存储当前不是必需项
- 队列当前还不是必须项，但如果搜索重建和同步开始变成真实后台任务，就应优先引入

## 9. 成功标准
第四版完成后，应满足以下标准：

1. 内容系统不再完全依赖静态构建时聚合
2. 后台可以查看和治理内容、索引、统计、任务状态
3. 搜索、统计、同步具备正式服务端边界
4. MySQL 与 Redis 至少完成一轮端到端运行验证
5. 第五版可以在第四版基础上继续做用户能力、开放能力或更复杂自动化
