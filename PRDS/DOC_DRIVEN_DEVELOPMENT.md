# 文档驱动开发协作规范

## 1. 文件用途

这份文档用于约束 `my-blog` 项目的日常协作方式。

目标不是“做完之后补文档”，而是：

- 先用文档定义范围
- 再按文档推进实现
- 实现后立即同步文档状态

以后每次开始新任务前，都应先阅读这份文件，再进入分析、设计、实现和收尾。

## 2. 核心原则

### 2.1 先文档，后实现

除非只是极小的错别字、格式修正或纯说明性调整，否则不要一上来直接改代码。

正常顺序必须是：

1. 先确认本次任务属于哪个功能模块
2. 先更新对应 PRD / DEV_PLAN / UI 文档
3. 再开始代码实现
4. 实现完成后，立即回写文档状态

### 2.2 总文档和子文档必须同时维护

不能只更新总文档，不更新功能子目录。

如果某个功能发生变化，至少要同时检查两层文档：

- 总文档
  - `PRDS/第一版/产品需求.md`
  - `PRDS/第一版/开发计划.md`
  - `PRDS/第一版/VERSION_NOTES.md`
- 功能子文档
  - `PRDS/第一版/01-首页/*`
  - `PRDS/第一版/02-博客系统/*`
  - `PRDS/第一版/03-笔记系统/*`
  - `PRDS/第一版/04-项目展示/*`
  - `PRDS/第一版/05-关于页/*`

### 2.3 文档必须反映“当前真实状态”

文档不能只是理想计划，也不能落后于代码现状。

每份文档都应该明确写出：

- 这个功能现在是未开始、进行中、已完成，还是暂缓
- 它是不是当前版本范围的一部分
- 它当前有没有正式实现
- 后续还缺什么

## 3. 开工前必须做的事

每次新任务开始时，先执行下面的判断：

### 3.1 判断任务类型

先判断这次任务属于哪种：

- 新功能
- 现有功能扩展
- 视觉优化
- 文档整理
- Bug 修复
- 范围调整

### 3.2 找到对应文档

根据任务所在模块，定位对应的文档目录。

例如：

- 首页：`PRDS/第一版/01-首页/`
- 博客系统：`PRDS/第一版/02-博客系统/`
- 笔记系统：`PRDS/第一版/03-笔记系统/`
- 项目展示：`PRDS/第一版/04-项目展示/`
- 关于页：`PRDS/第一版/05-关于页/`

### 3.3 先判断要不要改文档

只要任务会影响以下任一内容，就必须先改文档再改代码：

- 功能范围
- 页面结构
- 视觉方向
- 路由
- 内容模型
- 实现优先级
- 当前完成状态

## 4. 各类文档分别负责什么

### 4.1 `产品需求.md`

负责定义版本级的产品范围。

应该写：

- 当前版本目标
- 当前版本包含哪些功能
- 哪些功能明确不做
- 哪些功能暂缓但仍保留在本版本

不应该写：

- 过细的实现步骤
- 大量开发流水账

### 4.2 `开发计划.md`

负责定义版本级推进顺序和阶段状态。

应该写：

- 阶段划分
- 每阶段目标
- 当前完成到哪一步
- 第一版待补项
- 更适合放到第二版的内容

### 4.3 `VERSION_NOTES.md`

负责记录已经发生的迭代结果。

应该写：

- 本次完成了什么
- 本次明确没做什么
- 当前版本处于什么阶段
- 建议下一步做什么

### 4.4 功能目录下的 `PRD.md`

负责定义某个单独功能“是什么、为什么做、当前状态是什么”。

至少应该写清：

- 功能定位
- 当前版本状态
- 当前实现范围
- 暂不要求的内容

### 4.5 功能目录下的 `DEV_PLAN.md`

负责定义某个功能“接下来怎么做”。

至少应该写清：

- 当前状态
- 开发目标
- 建议实施顺序
- 完成标准

### 4.6 功能目录下的 `UI.md`

负责定义某个功能“视觉上想做成什么样”。

至少应该写清：

- 当前 UI 状态
- 视觉方向
- 页面气质
- 和其他页面的差异
- 后续优化方向

## 5. 具体执行顺序

每次任务默认按下面顺序推进：

1. 先阅读本文件
2. 阅读相关总文档
3. 阅读相关功能子文档
4. 判断本次任务会影响哪些文档
5. 先更新文档
6. 再做代码实现
7. 实现后更新状态文档
8. 最后汇报“改了什么、文档同步到哪、下一步是什么”

## 6. 哪些情况不能直接开写代码

遇到下面这些情况，禁止跳过文档直接开发：

- 新增页面
- 新增路由
- 调整版本范围
- 把某个功能从“待补”改成“开始实现”
- 明显改变页面结构
- 明显改变视觉方向
- 引入新的内容模型

## 7. 哪些情况可以先实现后补极少量文档

只有以下这种轻量情况，可以不把流程搞得很重：

- 错别字修正
- 样式微调且不影响结构
- 已有功能的小 Bug 修复
- 文案替换但不影响页面结构

即使是这些情况，做完后也要判断是否需要补一句状态说明。

## 8. 状态词统一规范

为了避免文档里口径混乱，统一使用下面几种状态：

- `未开始`
- `进行中`
- `已完成`
- `第一版待补项`
- `暂缓`
- `已在第一批正式实现中完成`

不要一会儿写“以后再说”，一会儿写“先不做”，一会儿写“第二版再看”，要尽量统一。

## 9. 版本划分规则

如果某个功能这次没做，不代表它自动进入第二版。

优先按下面规则判断：

- 如果它本来就在第一版目标里，只是当前优先级较低：写成“第一版待补项”
- 如果它属于真正的新能力、新复杂度或新系统：再考虑放入第二版

例如：

- `notes`、`projects` 更适合写成第一版待补项
- 搜索、评论、数据库、后台更适合定义为第二版方向

## 10. 本项目当前默认工作方式

针对 `my-blog`，后续默认采用下面的任务启动方式：

1. 先读 `PRDS/DOC_DRIVEN_DEVELOPMENT.md`
2. 再读相关 PRDS 文档
3. 先同步文档状态
4. 再进入正式实现

如果用户没有特别说明，也应该默认遵守这套流程，而不是盲目开始编码。

## 11. 给 AI 的直接指令模板

以后开始任务时，可以直接对 AI 说：

```md
先阅读 `PRDS/DOC_DRIVEN_DEVELOPMENT.md`，按文档驱动开发流程执行这次任务。
先同步相关 PRDS 文档，再开始实现。
```

如果任务只涉及某个模块，也可以说：

```md
先阅读 `PRDS/DOC_DRIVEN_DEVELOPMENT.md`。
这次任务只针对 `PRDS/第一版/02-博客系统/`，先同步该模块文档，再开始代码修改。
```

## 12. 收尾要求

每次任务结束时，必须明确回答：

- 这次改了哪些代码或文档
- 哪些文档已同步
- 当前状态是未开始、进行中还是已完成
- 下一步最合理做什么

如果文档没有同步，不能把任务算作完整完成。
